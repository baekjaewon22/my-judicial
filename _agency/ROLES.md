# 권한 매트릭스

> 앱 내부의 역할(role)과 각 역할이 수행 가능한 액션.
> 이 표는 코드(`src/types/roles.ts`, 라우트 가드, UI 조건부 렌더)와 반드시 일치해야 합니다.
> 변경 시 `/sync-context` 로 일관성 점검.

## 역할 정의

| 역할 키 | 한글 명칭 | 설명 |
|---|---|---|
| `admin` | 대표/시스템관리자 | 모든 권한 |
| `accountant` | 총무담당 | 매출·정산 실무 |
| `accountant_asst` | 총무보조 | 입력만, 변경은 승인 필요 |
| `viewer` | 열람자 | 조회만 |

|---|:---:|:---:|:---:|:---:|

| 환불 처리 | ✅ | 🔸승인 | ❌ | ❌ |
| 알림톡 발송 | ✅ | ✅ | 🔸승인 | ❌ |
| 사용자 관리 | ✅ | ❌ | ❌ | ❌ |
| 권한 변경 | ✅ | ❌ | ❌ | ❌ |

범례: ✅ 허용 · ❌ 거부 · 🔸 승인 필요(pending_approvals 테이블)

## 승인 게이트 동작

🔸 표시된 액션 처리 절차:

1. 요청자가 액션 실행 → 시스템이 `pending_approvals` 테이블에 기록
2. `admin` 또는 상위 권한자에게 알림
3. 승인 시 원 요청이 실행, 거부 시 사유와 함께 기록
4. 모든 승인·거부는 `activity_log` 에 영구 보존

## 코드 일관성 체크포인트

`/sync-context` 실행 시 다음이 일치하는지 검사:

1. `src/types/roles.ts` 의 유니언 타입 ↔ 이 표의 역할 키
2. 라우트 가드 (`requireRole`) ↔ 이 표의 허용 권한
3. 프론트엔드 UI 노출 조건 ↔ 이 표
4. 마이그레이션의 `roles` 컬럼 체크 제약 ↔ 이 표

## 변경 프로토콜

권한 매트릭스 변경 시:

1. 이 표를 먼저 수정
2. ADR 작성 (`docs/decisions/NNNN-role-change.md`)
3. `src/types/roles.ts` 업데이트
4. 라우트 가드·UI 동기화
5. 마이그레이션 필요 시 추가
6. `/sync-context` 로 최종 검증
